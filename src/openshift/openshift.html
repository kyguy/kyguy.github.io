Overview

    Openshift is a container platform which facilitates the development and management of software applications across distributed systems. Whether the components of these distributed systems be physical hosts, virtual machines or a combination of both, OpenShift reaps the benefits of available computing resources without the complexity typically tied with computing power.


Figure 1: OpenShift Cluster

Core Components

    The OpenShift tools and concepts help plan how an application should behave when its online: when it should scale, where it should save its data, how many application backups are benched, etc. Then, the OpenShift application carries out the plan’s execution.

Pods

To run on OpenShift, software applications must be formatted onto a virtual host known as a pod. This virtual compute unit is essentially just a group of containers or isolated linux processes, that can share disk and networking space. At a higher level of abstaction, a pod acts as a single computer node, where applications can be built and hosted.


Figure 2: Anatomy of a Pod

Pods provide the advantage of isolating applications running across a cluster. This helps better utilize compute nodes and avoid environment problems like conflicting libraries or reused port numbers. However, the core strength of pods revolves arounds OpenShift’s ability to manipulate them. Once in a container format, a pod application can be updated once and have its changes automatically distributed amongst the nodes running in a cluster.

Images

An image is a snapshot of a container that can be instantiated to create a new containers.


Figure 3: Registries and Images

API Server

All components of an OpenShift cluster, internal and external, communicate through a Hypertext Transfer Protocol (HTTP) interface. Within every OpenShift cluster resides one application programming interface (API) server which interprets the HTTP commands and executes the respective operations server side.


Figure 4: API Server

This means that any programming language can manipulate an OpenShift cluster so long as that language can send HTTP requests. However, OpenShift already provides client tools for common operations. In the following example, the client command, (oc whoami -t), is used to retrieve the authorization token to take a look at the HTTP interface, or Representational State Transfer (REST) API:
